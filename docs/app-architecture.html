SCHNGN architecture

Local-first Schengen planner on Cloudflare Workers

The important line: guest trip dates stay in the browser. A clearly labelled signup-and-save action uses Clerk for account creation, then Cloudflare stores the authenticated account data. The calculation engine stays pure and testable.

Safety-critical core

The 90/180 calculation engine is isolated in packages/engine and must be tested before UI polish outruns correctness.

Privacy boundary

Guest trips stay in browser storage. Choosing signup-and-save explicitly starts Clerk account creation, after which current trips sync to separate D1 rows; trip data never enters analytics or logs.

Runtime boundary

Bun and Node 24 are tooling. Production is Cloudflare workerd plus static assets; deployed code avoids Bun-native APIs.
